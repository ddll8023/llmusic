# 编程工作流

> 本文档定义了 Claude Code 的标准编程工作流，包含任务分类、思考链、阶段识别和 GAN 对抗评估机制。

**适用范围：** 所有代码实现、调试修复、功能优化任务

---

## 一、工作流概述

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| **任务驱动** | 根据任务类型自动路由到对应思考链 |
| **阶段分离** | 方案设计与代码执行严格分离 |
| **安全确认** | 任何代码修改必须收到明确安全词 |
| **客观评估** | GAN 对抗模式确保评估独立性 |
| **全面分析** | 评估覆盖问题、方案、代码三个层次 |

### 1.2 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      用户输入                                    │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│  【阶段一】任务分类                                              │
│  ├─ 分析用户意图                                                │
│  ├─ 判断任务类型（8大类）                                        │
│  └─ 路由到对应的思考链                                          │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│  【阶段二】思考链执行                                            │
│  ├─ 读取相关文档（doc/ 和 规范文档/）                            │
│  ├─ 执行类型特定思考链                                          │
│  ├─ 调用对应 Skill                                             │
│  └─ 生成分析报告和方案                                          │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│  【阶段三】阶段自动识别 ⚠️                                       │
│  ├─ 方案设计阶段 → 展示方案，等待讨论                            │
│  ├─ 等待确认阶段 → 等待安全词"嘻嘻"                             │
│  └─ 用户提出修改 → 返回阶段二                                    │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│  【阶段四】代码执行                                              │
│  ├─ 收到"嘻嘻"后执行                                            │
│  ├─ 按方案修改代码                                              │
│  └─ 生成修改记录                                                │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│  【阶段五】GAN 对抗评估（独立 Agent）                            │
│  ├─ 第一层：问题分析评估                                         │
│  ├─ 第二层：方案完整性评估                                       │
│  └─ 第三层：代码质量评估                                         │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
              ┌─────────────┴─────────────┐
              ↓                           ↓
          ✅ 通过                      ❌ 不通过
          声称完成                      返回修复
```

### 1.3 阶段概览

| 阶段 | 触发条件 | 行为 | 等待确认？ |
|------|---------|------|-----------|
| 阶段一：任务分类 | 用户输入 | 自动分析并路由 | 否 |
| 阶段二：思考链执行 | 任务类型已确定 | 调用 Skill，生成方案 | 展示方案后等待 |
| 阶段三：阶段识别 | 方案已展示 | 判断是讨论还是等待 | 等待"嘻嘻" |
| 阶段四：代码执行 | 收到"嘻嘻" | 执行代码修改 | 否 |
| 阶段五：GAN 评估 | 代码已修改 | 独立 Agent 审查 | 否 |

## 二、任务分类矩阵

### 2.1 八大任务类型

| 类型 | 标识词 | 描述 | 对应 Skill |
|------|--------|------|-----------|
| **代码实现** | 添加、实现、新增、建立、创建 | 新功能开发 | test-driven-development |
| **调试修复** | 修复、bug、错误、异常、失败、问题 | 缺陷修复 | systematic-debugging |
| **前端 UI** | 界面、页面、组件、样式、布局、颜色、动效 | 界面开发/优化 | frontend-design + 对应子技能 |
| **代码质量** | 优化、重构、简化、清理、精简 | 代码质量提升 | simplify / harden / distill |
| **审查评审** | 审查、review、评审、检查 | 代码审查 | requesting-code-review |
| **系统配置** | 配置、环境、设置、权限 | 环境/配置变更 | update-config |
| **用户体验** | 体验、交互、引导、空状态 | UX 优化 | onboard / critique / clarify |
| **技术审计** | 审计、性能、无障碍、安全 | 技术评估 | audit |

### 2.2 路由规则

```
用户输入
  ↓
提取关键词
  ↓
匹配任务类型（按顺序检查）
  ↓
├─ 包含"修复/bug/错误" → 调试修复
├─ 包含"添加/实现/新增" → 代码实现
├─ 包含"界面/页面/样式/布局" → 前端 UI
├─ 包含"优化/重构/简化" → 代码质量
├─ 包含"审查/review" → 审查评审
├─ 包含"配置/环境/设置" → 系统配置
├─ 包含"体验/引导/空状态" → 用户体验
├─ 包含"审计/性能/安全" → 技术审计
└─ 无法识别 → 默认代码实现 + 追问澄清
```

### 2.3 类型特定 Skill 组合

**代码实现：**
```
test-driven-development
  ↓（如需前端）
+ frontend-design
```

**调试修复：**
```
systematic-debugging
  ↓（如需质量检查）
+ simplify
```

**前端 UI：**
```
frontend-design
  + arrange（布局问题）
  + colorize（颜色问题）
  + typeset（字体问题）
  + animate（动效问题）
  + optimize（性能问题）
```

**代码质量：**
```
simplify（重复/冗余）
harden（边界/异常）
distill（功能太多）
normalize（样式漂移）
polish（细节打磨）
```

## 三、思考链体系

### 3.1 通用思考链（所有任务必须执行）

```
═══════════════════════════════════════════════════════════════
  🔍 通用思考链
═══════════════════════════════════════════════════════════════

【第零层】项目上下文准备 ⚠️ 必须先执行
├─ 读取 doc/ 目录下的相关文档（了解项目背景、技术方案）
├─ 读取 规范文档/ 目录下的相关文档（了解开发规范）
└─ 然后才能进入后续思考链

【第一层】理解用户意图
├─ 用户真正想要什么？
├─ 问题的本质是什么？
└─ 追问澄清（如有必要）

【第二层】确认任务类型
├─ 根据关键词识别任务类型
├─ 选择主要 Skill
└─ 如有需要，选择辅助 Skill

【第三层】执行类型特定思考链
└─ （见 3.2 节）

【第四层】生成分析报告
├─ 问题分析
├─ 相关文件列表
├─ 实现方案
└─ 风险识别

═══════════════════════════════════════════════════════════════
```

### 3.2 类型特定思考链

#### 3.2.1 代码实现思考链

```
第三层-1：问题分析
├─ 需要实现什么功能？
├─ 功能的边界是什么？
└─ 有哪些前置条件？

第四层：影响分析 ⚠️ 重要
├─ 需要修改哪些文件？
├─ 是否需要新增文件？
├─ 是否有关联模块需要同步修改？
└─ 是否会影响现有功能？

第五层：方案设计
├─ 数据层：DTO、Mapper、Entity
├─ 业务层：Service
├─ 接口层：Controller、接口文档
└─ 依赖关系

第六层：风险识别
├─ 并发问题
├─ 数据一致性
└─ 兼容性
```

#### 3.2.2 调试修复思考链

```
第三层-1：问题定位
├─ 复现问题
├─ 定位问题代码
└─ 确定问题根因

第四层：影响分析 ⚠️ 重要
├─ 问题的范围有多大？
├─ 哪些模块/功能受影响？
├─ 是否存在类似问题？
└─ 相关文件是否都考虑到了？

第五层：修复方案
├─ 临时修复 vs 根本修复
├─ 修复的副作用
└─ 是否需要回归测试

第六层：验证方案
├─ 如何验证修复有效？
└─ 需要哪些测试用例？
```

#### 3.2.3 前端 UI 思考链

```
第三层-1：需求分析
├─ 需要什么样的界面效果？
├─ 参考案例？
└─ 响应式要求？

第四层：影响分析 ⚠️ 重要
├─ 需要修改哪些组件？
├─ 是否需要新增组件？
├─ 样式是否影响其他页面？
└─ 相关页面是否需要同步调整？

第五层：方案设计
├─ 组件结构
├─ 样式方案
├─ 交互逻辑
└─ 状态管理
```

#### 3.2.4 代码质量思考链

```
第三层-1：问题识别
├─ 存在什么质量问题？
├─ 问题代码的位置？
└─ 问题的严重程度？

第四层：影响分析 ⚠️ 重要
├─ 修改会影响哪些调用方？
├─ 是否有依赖关系需要处理？
├─ 重构风险有多大？
└─ 相关代码是否都需要处理？

第五层：重构方案
├─ 重构步骤
├─ 向后兼容
└─ 测试策略
```

## 四、阶段自动识别

### 4.1 阶段定义

| 阶段标识 | 描述 | 等待行为 |
|---------|------|---------|
| **方案设计** | 正在分析问题、设计方案 | 等待用户讨论或确认 |
| **等待确认** | 方案已展示，等待安全词 | 等待"嘻嘻" |
| **代码执行** | 执行代码修改 | 不等待 |
| **评估反馈** | 评估不通过，等待修复 | 等待新方案 |

### 4.2 阶段识别规则

```
代码修改状态
  ↓
├─ 尚未展示方案 → 【方案设计阶段】
│                    ↓
│               展示方案
│                    ↓
│            【等待确认阶段】
│                    ↓
│   ┌────────────────┼────────────────┐
│   ↓                ↓                ↓
│ 用户提出修改    用户说"同意"    用户说"嘻嘻"
│   ↓                ↓                ↓
│【方案设计阶段】 【等待确认阶段】 【代码执行阶段】
│ (重新设计)     (继续等待)       (执行代码)
│
└─ 代码已修改 → 【评估阶段】
                    ↓
         ┌──────────┴──────────┐
         ↓                     ↓
     ✅ 评估通过             ❌ 评估不通过
         ↓                     ↓
     【完成】              【方案设计阶段】
                              (修复问题)
```

### 4.3 安全词机制

**安全词定义：**

| 词汇 | 行为 | 说明 |
|------|------|------|
| `"嘻嘻"` | **执行代码修改** | 唯一触发代码执行的词汇 |
| `"好的"/"可以"/"同意"/"没问题"` | 确认方案，但不执行 | 继续等待"嘻嘻" |
| 其他内容 | 继续讨论 | 视为讨论或修改意见 |

### 4.4 阶段行为对照表

| 用户输入 | 当前阶段 | 识别结果 | 下一步行为 |
|---------|---------|---------|-----------|
| 展示方案 | - | 方案设计 | 进入等待确认阶段 |
| "可以/好的/同意" | 等待确认 | 确认方案 | 继续等待"嘻嘻" |
| "嘻嘻" | 等待确认 | **执行代码** | 进入代码执行阶段 |
| 提出修改意见 | 等待确认 | 方案设计 | 返回阶段二，重新设计 |
| 提出新需求 | 任意 | 方案设计 | 进入新任务的阶段一 |

## 五、GAN 对抗评估

### 5.1 评估架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        主 Agent                                 │
│  （业务逻辑 Agent，负责问题分析和代码实现）                       │
│                                                                 │
│  输入：用户需求                                                  │
│  输出：分析报告 + 代码修改                                       │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 分析报告内容：                                            │   │
│  │ - 问题描述                                               │   │
│  │ - 问题根因                                               │   │
│  │ - 相关文件列表（所有涉及的文件）                          │   │
│  │ - 实现方案                                               │   │
│  │ - 风险识别                                               │   │
│  └─────────────────────────────────────────────────────────┘   │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
                     分析报告 + 代码
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│                    Reviewer Agent                               │
│  （独立评估 Agent，不知道主 Agent 的设计意图）                   │
│                                                                 │
│  输入：分析报告 + 代码 + 项目规范                                │
│  输出：评估报告                                                  │
│                                                                 │
│  ⚠️ 评估前不知道：                                               │
│  - 主 Agent 的设计方案是什么                                     │
│  - 主 Agent 识别了哪些相关文件                                   │
│  （盲评模式，确保客观性）                                        │
└───────────────────────────┬─────────────────────────────────────┘
                            ↓
                       评估报告
                            ↓
              ┌─────────────┴─────────────┐
              ↓                           ↓
          ✅ 通过                      ❌ 不通过
          声称完成                      返回修复
```

### 5.2 第一层：问题分析评估

**评估目标：** 验证主 Agent 对问题的理解是否正确、全面。

**评估维度：**

| 评估项 | 检查点 | 评分 |
|--------|--------|------|
| 问题描述 | 是否准确描述了用户的需求 | 0-3 |
| 问题边界 | 是否清晰界定了问题范围 | 0-3 |
| 根本原因 | 是否识别了问题的根本原因（调试场景） | 0-3 |
| 追问澄清 | 是否在必要时追问了用户 | 0-3 |

**评估标准：**

| 评分 | 等级 | 处理方式 |
|------|------|---------|
| 9-12 | ✅ 通过 | 继续下一层评估 |
| 6-8 | ⚠️ 警告 | 记录警告，继续评估 |
| 0-5 | ❌ 不通过 | 返回修复，问题分析阶段 |

**评估检查单：**
```
□ 问题描述是否完整？
□ 问题边界是否清晰？
□ 是否遗漏了隐含需求？
□ 是否识别了 stakeholders（调用方、依赖方）？
□ 是否在必要时进行了追问？
```

### 5.3 第二层：方案完整性评估 ⚠️ 重点

**评估目标：** 验证主 Agent 的方案是否考虑了所有相关文件和关联模块。

**评估维度：**

| 评估项 | 检查点 | 评分 |
|--------|--------|------|
| 文件覆盖 | 相关文件是否都识别到了 | 0-3 |
| 关联分析 | 是否分析了模块间的关联关系 | 0-3 |
| 影响评估 | 是否评估了对现有功能的影响 | 0-3 |
| 风险识别 | 是否识别了潜在风险点 | 0-3 |

**评估标准：**

| 评分 | 等级 | 处理方式 |
|------|------|---------|
| 9-12 | ✅ 通过 | 继续下一层评估 |
| 6-8 | ⚠️ 警告 | 记录警告，继续评估 |
| 0-5 | ❌ 不通过 | 返回修复，补充相关文件 |

**评估检查单：**
```
□ 相关文件列表是否完整？
□ 是否识别了关联模块？
□ DTO/Entity 是否完整？
□ Service 层是否需要修改？
□ Controller 层是否需要调整？
□ Mapper/XML 是否需要同步？
□ 前端是否需要调整？
□ 配置是否需要更新？
□ 测试文件是否需要补充/修改？
□ 文档是否需要更新？
```

**特殊检查（调试场景）：**
```
□ 是否有其他类似问题需要一并处理？
□ 是否需要添加防护避免同类问题？
□ 是否需要回归测试？
```

### 5.4 第三层：代码质量评估

**评估目标：** 验证代码是否符合项目规范和质量标准。

**评估维度：**

| 评估项 | 检查点 | 评分 |
|--------|--------|------|
| 简洁性 | 无重复代码、无冗余逻辑 | 0-3 |
| 规范性 | 符合项目规范、代码风格 | 0-3 |
| 健壮性 | 边界处理、异常捕获、空指针 | 0-3 |
| 可读性 | 命名清晰、结构合理 | 0-3 |

**评估标准：**

| 评分 | 等级 | 处理方式 |
|------|------|---------|
| 9-12 | ✅ 通过 | 全部评估通过 |
| 6-8 | ⚠️ 警告 | 记录警告，评估通过 |
| 0-5 | ❌ 不通过 | 返回修复，代码质量问题 |

**评估检查单：**

**简洁性：**
```
□ 是否有重复代码可以提取？
□ 是否有冗余的中间变量？
□ 是否有不必要的嵌套？
□ 是否遵循 DRY 原则？
```

**规范性（后端）：**
```
□ DTO 是否添加 @Schema 注解？
□ 是否有必要的校验注解（@NotBlank、@Size 等）？
□ 是否有必要的注释？
□ 是否使用 @RequiredArgsConstructor？
□ 接口是否有 @Operation、@Parameter 注解？
□ Controller 是否只做请求转发？
```

**健壮性：**
```
□ 是否有空指针风险？
□ 是否有边界条件未处理？
□ 是否有异常捕获？
□ 是否有事务控制？
□ 是否有参数校验？
```

**可读性：**
```
□ 变量命名是否清晰？
□ 方法命名是否表达意图？
□ 代码结构是否清晰？
□ 是否有必要的分隔注释？
```

```markdown
# GAN 对抗评估报告

## 任务概览
- 任务类型：[类型]
- 评估时间：[时间]
- 评估 Agent：Reviewer

---

## 第一层：问题分析评估

| 评估项 | 得分 | 说明 |
|--------|------|------|

## 八、并行化执行规范

> 为提升工作效率，减少主 Agent 上下文消耗，引入多 Agent 并行处理机制。

### 8.1 并行化价值分析

```
┌─────────────────────────────────────────────────────────────────┐
│                      并行化收益对比                              │
├─────────────────┬─────────────────┬─────────────────────────────┤
│ 并行场景         │ 节省上下文       │ 风险等级                    │
├─────────────────┼─────────────────┼─────────────────────────────┤
│ 文档读取（3+文件）│ ~1500 tokens    │ 低（只读操作）              │
│ GAN评估（三层）  │ ~1800 tokens    │ 低（有标准约束）            │
│ 多文件独立修改   │ ~800 tokens/文件 │ 中（需依赖分析）           │
│ 多子任务分析     │ ~1000 tokens/任务│ 中（需上下文注入）         │
└─────────────────┴─────────────────┴─────────────────────────────┘
```

### 8.2 并行化判断标准

| 任务特征 | 可并行条件 | 处理方式 |
|---------|-----------|---------|
| 多个独立文件修改 | 无共享写入 | Agent 并行修改 |
| 多层评估 | 评估维度独立 | Agent 并行评估 |
| 多个独立子模块 | 无循环依赖 | subagent-driven-development |
| 存在依赖关系 | 依赖可排序 | 按依赖顺序串行 |
| 需要共享状态 | - | ❌ 不允许并行 |

### 8.3 并行化前提条件

**必须满足以下条件才能启动并行执行：**

```
□ 任务已拆分为独立子任务
□ 每个子任务的输入/输出已明确声明
□ 依赖关系已分析完毕（reads/writes/depends_on）
□ 上下文摘要已准备完成
□ 评估标准已确认
□ 冲突处理策略已确定
```

### 8.4 依赖分析规范

#### 8.4.1 依赖声明模板

#### 8.4.2 依赖冲突检测

| 冲突类型 | 检测规则 | 处理方式 |
|---------|---------|---------|
| 写-写冲突 | 两个任务写入同一文件 | ❌ 串行化 |
| 读-写冲突 | 任务B读取任务A写入的文件 | A先B后 |
| 循环依赖 | A依赖B，B依赖A | ❌ 串行化整个环 |
| 无冲突 | 无共享资源 | ✅ 允许并行 |

   │Agent L1 │         │Agent L2 │         │Agent L3 │
   │问题分析 │         │方案完整 │         │代码质量 │
   │评估     │         │性评估   │         │评估     │
   │         │         │         │         │         │
   │并行执行  │         │并行执行  │         │并行执行  │
   └────┬────┘         └────┬────┘         └────┬────┘
        │                   │                   │
        └───────────────────┼───────────────────┘
                            ↓
                    ┌───────────────┐
                    │   结果合并     │
                    │ 1. 检查一致性 │
                    │ 2. 分歧裁定   │
                    │ 3. 生成报告   │
                    └───────────────┘
```

### 8.7 异常处理机制

| 异常场景 | 检测方式 | 处理方式 |
|---------|---------|---------|
| 子任务超时 | TaskOutput 超时 | 主 Agent 接管，或重新调度 |
| 子任务失败 | 返回 error | 分析原因，修复后重试该子任务 |
| 结果冲突 | 文件内容比对 | 主 Agent 裁定，保留更严格的方案 |
| 评估分歧 | 评分差异 > 2 | 主 Agent 复核，必要时二次评估 |
| 依赖分析错误 | 任务执行报错 | 回退串行模式，重新分析依赖 |

### 8.8 并行执行检查清单

**启动并行任务前必须确认：**

```
□ 依赖关系已明确（reads/writes/depends_on）
□ 无循环依赖
□ 无写-写冲突
□ 上下文已准备完整
□ 每个子任务有明确的输出格式要求
□ 异常处理策略已确定
□ 结果合并方案已设计
```

**并行任务完成后必须验证：**

```
□ 所有子任务已成功完成
□ 结果已按格式收集
□ 文件修改无冲突
□ 集成后功能正常
□ 评估已完成且通过
```

### 8.9 适用场景速查

| 场景 | 推荐方式 | 说明 |
|------|---------|------|
| 读取多个文档 | 并行 Read | 无冲突，只读操作 |
| GAN 三层评估 | subagent-driven-development | 独立评估维度 |
| 修改多个无关文件 | Agent 并行 | 需先确认无依赖 |
| 复杂任务多子模块 | subagent-driven-development | 需依赖分析 |
| 有依赖关系的任务 | 串行 | 必须按依赖顺序 |
